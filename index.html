01
Context & Objective
Simulated a real-world SOC deployment for Catnip Games International — a fictional games company preparing for its first multiplayer launch. As Team Lead, I owned TheHive end-to-end and coordinated integrations across a 5-person team spanning MISP, Cortex, automation scripting, documentation, and KPIs. The goal was a fully operational, integrated SOC platform running on a shared gateway VM (192.168.200.1) with automated alert triage and case management.
Architecture & Infrastructure
  • Gateway VM (192.168.200.1) — single Linux host running TheHive 5.6, Cortex, MISP (via apache2), Cassandra, and Elasticsearch in a defined startup order to avoid dependency failures
  • TheHive 5.2 → 5.6 upgrade performed mid-project to enable full Cortex integration; required Cassandra and Elasticsearch reconfiguration
  • Startup sequence: cassandra (sleep 15s) → elasticsearch (sleep 10s) → thehive, cortex, apache2 — sequence enforced to prevent service crash-on-boot
  • RBAC configured for analyst, manager, and admin roles with custom case templates and dashboards per role
Integrations Built
  • MISP → TheHive: threat intelligence feeds connected, IOC tags syncing bi-directionally; threat data populates directly into active cases
  • Cortex → TheHive: analysers configured including VirusTotal, AbuseIPDB, URLScan, MaxMind — analysts trigger enrichment from within a case with one click
  • Cortex-Analyzers cloned to /opt/Cortex-Analyzers/; configured with individual API keys per service

Multi-VM Enterprise Network Security Sandbox
Networking & Security Practice Module · University of Roehampton · 2026
04
Objective
Built a 4-VM VirtualBox lab simulating a segmented enterprise network — Gateway, Workstation, Web Server, and Zabbix Monitoring Server — to practice network security hardening, traffic analysis, and monitoring configuration from the ground up.
Network Topology & Configuration
  • Gateway VM: iptables firewall rules enforcing network segmentation; NAT + host-only adapter configuration; blocked inter-VM traffic except defined service ports
  • SSH hardening: key-based authentication only (PasswordAuthentication disabled); port changed from 22; MaxAuthTries set; fail2ban equivalent monitoring configured for failed login lockout
  • Zabbix monitoring server: configured agents on all VMs; dashboards for CPU, memory, network traffic; security event detection rules for failed SSH attempts and unusual outbound connections
  • Web server: Apache with minimal exposed services; headers stripped to prevent version disclosure
Security Testing & Analysis
  • Nmap reconnaissance: scanned internal network topology to verify firewall rules blocked expected ports; OS fingerprinting used to confirm hardening impact on banner disclosure
  • Wireshark packet capture: analysed traffic between VMs to verify no unintended lateral movement paths; confirmed SSH traffic encryption and absence of plaintext credentials
  • Documented via video demonstration — full setup walkthrough and security configuration evidence for academic assessment
Demonstrated end-to-end network security architecture: segmentation enforced at gateway, hardened access controls, and live monitoring — all validated through active reconnaissance of the lab environment.

Malware Analysis — Isolated Sandbox Environment
Academic Coursework · Queen's University Belfast · 2024
06
Objective
Conducted both static and dynamic analysis of malicious binary samples in an isolated sandbox environment to understand behaviour, extract indicators of compromise, and document findings for threat detection use — simulating the workflow of a malware analyst in a SOC or DFIR context.
Static Analysis
  • IDA Pro disassembly: decompiled binary to identify function calls, API imports, and string artefacts without execution
  • String extraction: identified hardcoded C2 domain indicators, registry key paths, and mutex names embedded in the binary
  • PE header analysis: section entropy checked for packing — high entropy in .text section indicated potential packer; import table analysed for suspicious API calls (VirtualAlloc, CreateRemoteThread, WriteProcessMemory)
Dynamic Analysis & IOCs Extracted
  • Process behaviour: monitored spawned child processes and injected threads; identified persistence via Run key modification on first execution
  • Network communication: DNS queries captured for C2 domains; HTTP beacon pattern identified at regular interval
  • File system activity: dropper wrote secondary payload to %TEMP%; original binary deleted post-execution
  • IOC set produced: SHA-256 file hash, mutex name, C2 domain/IP, registry persistence key, dropped file path — formatted for SIEM ingestion
Full IOC set documented and formatted for threat intelligence sharing. Analysis workflow mirrors real-world DFIR triage — static first to avoid detonation risk, dynamic in isolated VM to confirm behaviour.

Cloud-Native NP-Complete Solver — Distributed Sudoku Framework
Final Year Project · Queen's University Belfast · 2024–2025 · github.com/Prabhath77 ↗
08
Problem & Approach
Designed and built a cloud-native distributed system to solve Sudoku as a representative NP-complete constraint satisfaction problem. The goal was to demonstrate that horizontal scaling via containerisation could provide meaningful performance gains over single-node solving — implemented with a Master-Worker architecture on Docker and Kubernetes.
Architecture
  • Master node: receives puzzle input, partitions the constraint space, distributes sub-problems to available workers, aggregates solutions
  • Worker nodes: containerised Node.js processes running assigned solver algorithm; communicate results back to master via message passing
  • Kubernetes orchestration: horizontal pod autoscaling; worker replicas scale with problem complexity; health checks and restart policies configured
Algorithms Benchmarked
  • Backtracking: baseline recursive DFS — exponential worst-case but simple to partition across workers
  • Constraint propagation (Arc Consistency): reduces search space before backtracking — significant speedup on sparse puzzles
  • Heuristic search (MRV — Minimum Remaining Values): selects the most constrained variable first — best performance on hard instances
Demonstrated measurable speedup with distributed solving compared to single-node baseline — with MRV heuristic achieving best performance across all puzzle difficulty categories. Codebase publicly available on GitHub.
